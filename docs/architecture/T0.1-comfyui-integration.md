This document is the canonical integration reference for the Comfy Mobile client. It is updated incrementally; section status flags below indicate maturity.

## 0. Scope (post-decision narrowing 2026-05-10)

In scope:
- HTTP REST surface, WebSocket events, workflow JSON shape, output retrieval, queue & history semantics, auth-free LAN.
- Manual `host:port` connection workflow, history list, reachability probing, error categorization.

## 1. HTTP REST surface  *(Status: M1 ✅)*

ComfyUI's HTTP API runs on port 8188 by default (aiohttp). Endpoints used by mobile client are noted with their need-mapping (1=Import, 2=Edit-workflow, 3=Edit-nodes, 4=Generate/output, 5=History).

| Method | Path | Need | Purpose |
|---|---|---|---|
| GET | `/system_stats` | — | Liveness + RAM/VRAM/version probe (use as connect health-check). |
| GET | `/features` | — | Server feature flags (e.g. supports_preview_metadata). Read once at connect. |
| GET | `/object_info` | 1,2,3 | Full node-class catalog incl. inputs/outputs/defaults/required types. Must cache; large payload. |
| GET | `/object_info/{node_class}` | 3 | Single node-class detail; for incremental fetch when whitelist evolves. |
| GET | `/embeddings` | 2,3 | List of available embedding filenames (for prompt autocomplete). |
| GET | `/models` | 2 | Top-level model folder names. |
| GET | `/models/{folder}` | 2 | Files under a model folder (e.g. `checkpoints`, `loras`). Drives parameter pickers. |
| POST | `/upload/image` | 1,2 | Upload reference / input image. FormData: `image`, `type` (input/temp/output), `subfolder`, `overwrite`. |
| POST | `/upload/mask` | 2 | Upload mask overlay for an existing image. |
| POST | `/prompt` | 4 | **Submit workflow for execution.** JSON body: `{prompt, prompt_id?, number?, front?, extra_data?, client_id, partial_execution_targets?}`. Returns `{prompt_id, number, node_errors}`. |
| GET | `/prompt` | 4 | Current queue counters. |
| GET | `/queue` | 4 | Returns running & pending queue items with their prompts. |
| POST | `/queue` | 4 | `{clear: true}` empties queue; `{delete: [prompt_ids]}` removes specific items. |
| POST | `/interrupt` | 4 | Cancels current execution; optional `{prompt_id}` payload. |
| GET | `/history` | 5 | Paginated execution history. Query: `max_items`, `offset`. |
| GET | `/history/{prompt_id}` | 4,5 | Full history entry for a specific prompt — incl. outputs map and status. |
| GET | `/view` | 4,5 | **Output / preview asset retrieval.** Query: `filename`, `type` (input/temp/output), `subfolder`, `preview` (jpeg/webp;quality), `channel` (rgb/a). Streams raw bytes. |
| GET | `/view_metadata/{folder}` | 2 | Safetensors metadata (rare; for advanced model inspection). |
| GET | `/extensions` | — | JS extension files (frontend only — ignore on mobile). |

**Key implications for mobile client:**
- `/object_info` payload is **large** (every node class) and must be lazily cached on disk (`commonMain` cache layer). Refresh on server-version change.
- `/view` is the only image-fetch path. Outputs come back as `{filename, subfolder, type}` triples — mobile constructs full URL = `http://host:port/view?filename=...&subfolder=...&type=output`.
- `/prompt` POST requires **API-format workflow JSON**, not UI-format. See §3.

## 2. WebSocket events  *(Status: M1 ✅)*

Endpoint: `GET /ws?clientId={uuid}`. Mobile generates a stable client-id (UUID v4 persisted in app preferences) and reconnects with the same id; server scopes some events to that client.

### Server → client events

| Event | Trigger | Payload fields | UX use |
|---|---|---|---|
| `status` | On connect / queue change | `status: {exec_info: {queue_remaining}}`, `sid` | Shows "queue: N" badge; `sid` is the assigned client_id when none was supplied. |
| `feature_flags` | On connect | `flags: {...}` | Capability negotiation (we're consumer-side; just record). |
| `execution_start` | Workflow begins | `prompt_id` | "Generation started" UX state. |
| `execution_cached` | After cache-resolve | `nodes: [node_ids]`, `prompt_id` | These nodes will not emit `executing/executed`. Mobile pre-marks them as done in progress UI. |
| `executing` | A node begins | `node`, `display_node`, `prompt_id` | Current-node label / per-node spinner. `node=null` indicates "no node currently running" (idle/end). |
| `progress` | Per-step within a sampler node | `value`, `max`, `prompt_id`, `node` | Step-level progress bar. Most useful event for "X / Y steps" UX. |
| `progress_state` | Aggregate progress snapshot | `prompt_id`, `nodes: { node_id: {value,max,state} }` | Resume-after-reconnect snapshot. |
| `executed` | A node finishes | `node`, `display_node`, `output: {...}`, `prompt_id` | If `output.images` present, immediately fetchable via `/view`. Final node executed = generation done. |
| `execution_error` | Node throws | `prompt_id`, `node_id`, `node_type`, `executed: [completed_nodes]`, `exception_message`, `exception_type`, `traceback`, `current_inputs`, `current_outputs` | Show error sheet; offer "View details" with traceback (advanced). |
| `execution_interrupted` | User cancelled | `prompt_id`, `node_id`, `node_type`, `executed: [...]` | Confirms interrupt landed; UX can clear in-flight state. |
| `execution_success` | Workflow done | `prompt_id` | Final "done" signal. Authoritative — should drive the success UX over inferring from `executed`. |

### Client → server events

| Event | Payload | Use |
|---|---|---|
| `feature_flags` | `flags: {...}` | Sent right after connect to declare client capabilities. |

### Reconnection contract (drives T0.4 wait/abort UX)

After a dropped WS:
1. Reconnect with **same** `clientId`.
2. On `status` event we get fresh `queue_remaining`. To recover in-flight prompt state, hit `GET /history/{prompt_id}` for any prompts the user has in flight; if not yet completed, watch `progress_state` events going forward.
3. We cannot replay events that were missed during disconnect — design must rely on:
   - `progress_state` snapshots (pulls authoritative current state).
   - `/queue` HTTP poll as a fallback to detect "still running" vs "completed while we were offline".
   - `/history` to fetch any output produced during the offline window.

## 3. Workflow JSON: API format vs UI format  *(Status: M2 ✅)*

### TL;DR

- **API format**: minimal graph that `/prompt` accepts. `{node_id: {class_type, inputs}}`. **No** node titles, positions, link metadata, groups, or viewport.
- **UI format**: full editor save (the "workflow.json" the desktop UI exports without the "API format" toggle). Contains `nodes[]`, `links[]`, `groups`, viewport, version. Used by the desktop frontend; **not directly accepted by `/prompt`**.
- Frontend converts UI → API before submit. Mobile must do the same.

### API format — exact shape

```json
{
  "3": {
    "class_type": "KSampler",
    "inputs": {
      "seed": 8566257,
      "steps": 20,
      "cfg": 8.0,
      "sampler_name": "euler",
      "scheduler": "normal",
      "denoise": 1.0,
      "model":         ["4", 0],
      "positive":      ["6", 0],
      "negative":      ["7", 0],
      "latent_image":  ["5", 0]
    }
  },
  "4": {
    "class_type": "CheckpointLoaderSimple",
    "inputs": { "ckpt_name": "v1-5-pruned-emaonly.safetensors" }
  }
}
```

- Keys are **string** node ids (typically numeric strings `"1"`,`"2"`,…).
- `inputs` value is either a literal (`number | string | bool | array | object`) or a 2-element link tuple `[source_node_id_string, source_output_index_int]`.
- Output indices are positional within the source node's declared outputs (per `/object_info`).

### UI format — top-level shape

```json
{
  "last_node_id": 9,
  "last_link_id": 9,
  "nodes": [ /* see node entry below */ ],
  "links": [ /* link tuples */ ],
  "groups": [ /* visual group boxes */ ],
  "config": { /* misc UI config */ },
  "extra": { /* free-form metadata (often empty) */ },
  "version": 0.4
}
```

UI `nodes[]` entry:
```json
{
  "id": 3,
  "type": "KSampler",
  "pos": [863, 186],
  "size": {"0": 315, "1": 262},
  "flags": {},
  "order": 4,
  "mode": 0,
  "inputs":  [{"name": "model", "type": "MODEL", "link": 1}, ...],
  "outputs": [{"name": "LATENT", "type": "LATENT", "links": [7], "slot_index": 0}],
  "properties": {"Node name for S&R": "KSampler"},
  "widgets_values": [8566257, "randomize", 20, 8, "euler", "normal", 1],
  "title": "K Sampler"  // optional, user-renamed
}
```

UI `links[]` entry: `[link_id, src_node_id, src_slot, dst_node_id, dst_slot, type]`

### Conversion: UI → API (mobile must implement)

Pseudo-algorithm:
1. For each UI node, create entry `{class_type: node.type, inputs: {}}`.
2. For each declared input on the node:
   - If `input.link` is set, look up the link tuple → `inputs[input.name] = [String(src_node_id), src_slot]`.
   - If not linked, take the next consumed value from `widgets_values` according to the node's widget order (per `/object_info[node.type].input.required` ordering).
3. Drop `pos`, `size`, `order`, `mode`, `flags`, `properties`, `outputs`, `title`.
4. Drop top-level `groups`, `config`, viewport.

The widget-order ↔ widgets_values mapping is the trickiest part — it mirrors the order ComfyUI registers parameters in `INPUT_TYPES`. We **must** read `/object_info` to drive this for unknown nodes; for whitelist nodes we hard-code the ordering as a safety net.

### POST `/prompt` envelope

```json
{
  "prompt":        { /* API format */ },
  "client_id":     "<same uuid used on /ws>",
  "prompt_id":     "<optional uuid we choose>",
  "number":        0,        // optional priority
  "front":         false,    // optional: insert at queue front
  "partial_execution_targets": ["8"],  // optional: only run subset
  "extra_data": {
    "extra_pnginfo": {
      "workflow":  { /* original UI format JSON */ }
    },
    "api_key_comfy_org": "..."  // only if user uses Partner Nodes; not in v1
  }
}
```

**Mobile MUST set `extra_data.extra_pnginfo.workflow` to the *current* UI-format snapshot — i.e. the edited workflow at the moment of submission, NOT the originally imported JSON.** Otherwise a recipient who drags the generated PNG into desktop ComfyUI sees the un-edited workflow rather than what was actually run. If the workflow was imported as API-only (no UI metadata), synthesize a minimal UI snapshot (assign default positions, set `widgets_values` per descriptor, build `links[]` from API link tuples). Caught by @Lily during T0.6 CONTEXT.md review.

Response: `{ "prompt_id": "...", "number": N, "node_errors": { node_id: { errors: [...] } } }`. `node_errors` is non-empty when validation fails; mobile must surface these (likely as a non-modal "your workflow has issues" detail sheet).

### Storage strategy (mobile-side)

- Local store keyed by `(server_url, workflow_local_id)`:
  - `original_json` — verbatim (whichever format the user imported, untouched).
  - `format` — `"ui" | "api"`.
  - `derived_api_json` — recomputed on every edit.
  - `derived_ui_json` — kept in sync only if we have UI metadata; new UI metadata for parameter edits we make ourselves.
- Unknown nodes (not in descriptor table): render as collapsed pill, params read-only, JSON kept verbatim — they roundtrip safely because we never re-serialize them.

## 3.5 Node descriptor table v1 (skeleton — drives @Ores's T0.3)

This is the data-driven structure Ores's "whitelist" output will fill. T0.2 reserves the abstraction in `commonMain`. Each entry maps a `class_type` → display + editable params + control hints. **Selectable enum values come from `/object_info` at runtime** — we don't bake them.

Schema (Kotlin-pseudo):

```kotlin
data class NodeDescriptor(
    val classType: String,            // e.g. "KSampler"
    val displayName: LocalizedString, // i18n
    val category: String,             // "model", "prompt", "sampler", "latent", "decode", "save", "lora", "controlnet", "other"
    val editableParams: List<ParamDescriptor>,
    val helpText: LocalizedString? = null,
    val collapsedPreview: CollapsedPreview? = null  // what to show when collapsed in the graph
)

data class ParamDescriptor(
    val name: String,                 // param name in API inputs
    val displayName: LocalizedString,
    val control: ControlType,
    val source: ValueSource? = null,  // for pickers / autocomplete
    val constraints: Constraints? = null,
    val defaultBehavior: DefaultBehavior = DefaultBehavior.UseImported
)

sealed class ControlType {
    object Number : ControlType()                // single-line numeric
    object Integer : ControlType()
    data class Slider(val min: Double, val max: Double, val step: Double) : ControlType()
    object Toggle : ControlType()
    object SingleLineText : ControlType()
    object MultilineText : ControlType()
    object Dropdown : ControlType()              // values from source
    object ModelPicker : ControlType()           // values from source (server-known)
    object ImagePicker : ControlType()           // /upload/image flow
    object Hidden : ControlType()                // valid but not in mobile UI
}

sealed class ValueSource {
    object NodeEnumFromObjectInfo : ValueSource()                   // pull from /object_info[classType].input.required[name]
    data class ModelFolder(val folder: String) : ValueSource()      // pull from /models/{folder}
    object EmbeddingsList : ValueSource()                           // pull from /embeddings
}
```

### v1 entries for the 8 whitelist node families

> Notes: enum values for `sampler_name` / `scheduler` / etc come from `/object_info` at runtime, so `Dropdown` + `NodeEnumFromObjectInfo` is the canonical pattern.

| classType | displayName (zh / en) | editableParams |
|---|---|---|
| `CheckpointLoaderSimple` | 模型 / Checkpoint | `ckpt_name`: ModelPicker, source=ModelFolder("checkpoints") |
| `CLIPTextEncode` | 提示词 / Prompt | `text`: MultilineText (with embedding autocomplete via EmbeddingsList) |
| `KSampler` | 采样器 / Sampler | `seed`: Integer, `steps`: Slider(1..150, step 1), `cfg`: Slider(0..30, **step 0.5**), `sampler_name`: Dropdown (NodeEnumFromObjectInfo), `scheduler`: Dropdown (NodeEnumFromObjectInfo), `denoise`: Slider(0..1, **step 0.05**) |
| `EmptyLatentImage` | 画布尺寸 / Empty Latent | `width`: Slider(64..2048, step 64), `height`: Slider(64..2048, step 64), `batch_size`: Integer (1..16) |
| `VAEDecode` | 解码 / VAE Decode | (none — pure pass-through, but visible as pipeline step) |
| `SaveImage` | 保存 / Save | `filename_prefix`: SingleLineText |
| `LoraLoader` | LoRA | `lora_name`: ModelPicker (ModelFolder("loras")), `strength_model`: Slider(0..2, step 0.05), `strength_clip`: Slider(0..2, step 0.05) |
| `ControlNetLoader` | ControlNet | `control_net_name`: ModelPicker (ModelFolder("controlnet")) |

(Variants like `KSamplerAdvanced`, `CLIPTextEncodeSDXL`, `LoadImage` are deferred to v2 by Ores's whitelist; "unknown" rendering applies.)

### Unknown node handling

When `class_type` is **not** in the descriptor table:
- Display label = `class_type` (we get this directly from JSON; no need for `/object_info`).
- All params **read-only**, shown as a JSON-ish key/value list in a "details" expandable.
- Topology immutable in MVP per Lily's Q1 recommendation (see channel msg a3de4ec1) — no delete, no reconnect.
- Round-trip safe: we never modify the original entry.

## 4. Output / history semantics  *(Status: M1 ✅)*

- **Outputs are not pushed**. The server emits `executed` events containing per-node outputs, then the client fetches assets via `/view`.
- A typical SaveImage / PreviewImage node `output` payload looks like `{"images": [{"filename": "ComfyUI_00001_.png", "subfolder": "", "type": "output"}]}` (text/audio/video nodes have analogous shapes).
- `/history/{prompt_id}` returns a fully populated record after completion: `{prompt: [...], outputs: {node_id: {...}}, status: {status_str, completed, messages}}`. This is the source of truth for history view & background-recovery.
- Pagination: `/history?max_items=N&offset=K`. Note `/history` returns most-recent-first.

### Mobile cache strategy
- Maintain a local SQLite (SQLDelight) table `History(prompt_id, server_url, started_at, completed_at, status, ui_workflow_json, outputs_json)` keyed by `(server_url, prompt_id)`.
- Asset bytes (`/view` results) cached to disk LRU; thumbnails generated locally (Coil-multiplatform or platform-native).

## 5. Manual IP connection — engineering notes  *(Status: M1 ✅, replaces dropped mDNS branch)*

### Connect flow
1. User enters `host:port` (default `192.168.x.x:8188`).
2. Validate format (regex, RFC1123 host or IPv4/IPv6 literal, port 1–65535).
3. Hit `GET /system_stats` with short timeout (e.g. 3s). On 200 + parseable JSON → success; cache server signature `{name?, comfyui_version, python_version, gpus[]}`.
4. Subscribe to `/ws?clientId=<uuid>`.

### Error categorization (drives T0.4 error UX)
- **Format error** (regex fail) → inline form error, do not call network.
- **Timeout / refused** → "Server unreachable. Make sure ComfyUI is running and your phone is on the same Wi-Fi."
- **TLS / handshake** → "Couldn't establish a connection. Try without https." (we expect plain HTTP on LAN.)
- **Non-ComfyUI HTTP 200** → "Reached a server, but it doesn't look like ComfyUI." (system_stats parse fails.)
- **Wrong port / 404** → "ComfyUI says it's there, but the path is wrong." (system_stats returns 404.)

### History list
- Persist successful connections as `{label, host, port, last_connected_at}` in `commonMain` Settings (shared between platforms).
- On launch: if exactly one history entry, attempt auto-connect; show server picker if multiple.
- Friendly label per Ores: prompt user once after first successful connect.

### WiFi-only enforcement
- Android: `ConnectivityManager.getActiveNetwork()` + `NetworkCapabilities.hasTransport(TRANSPORT_WIFI)`; refuse connect on cellular with informational sheet.
- iOS: `NWPathMonitor` with `.usesInterfaceType(.wifi)`; same UX.

### Background suspension behavior
- Both iOS & Android suspend the WebSocket within ~30s of backgrounding (no special long-tasks declared). On foreground return:
  1. Detect last-known prompt(s) in flight.
  2. Open new WS with same `clientId`.
  3. For each in-flight `prompt_id`: `GET /history/{prompt_id}`. If completed → consume outputs + emit local "completed while away" notification. If still running → start subscribing to `progress`/`progress_state` going forward.
- For longer-running generations the user expects to continue, we accept that during suspension we have no precise progress; UX shows "running in the background — we'll update when you return" rather than fake live progress.

### Out-of-scope for v1 (note for future)
- mDNS / Bonjour auto-discovery (user request: deferred).
- Persistent foreground service (Android) / silent push (iOS) to keep WS open in background — adds significant complexity, defer until proven needed.

## 6. Three connection branches (handoff to T0.4)

For Ores's split-design intake:

### Branch A: Stable WebSocket
- Foreground use, on a healthy LAN. WS stays connected throughout generation.
- Fine-grained progress: `progress` event drives `value/max` step bar; `executing` highlights current node; `executed` paints per-node check.
- **UX ceiling**: per-step ETA, current-node label, cancel button is responsive (fires `/interrupt`).

### Branch B: Foreground but flaky LAN (Wi-Fi blip)
- WS drops mid-generation but server keeps running.
- Recovery via: same-clientId reconnect → next `progress_state` → fall back to `/history/{prompt_id}` poll if reconnect doesn't restore events within ~5s.
- **UX**: trust signal "still generating…" with an indeterminate spinner during reconnect window; switch back to step-bar once events resume.

### Branch C: Background suspension
- App suspended → WS gone → server still working → user re-foregrounds.
- On foreground: history poll resolves outcome. May arrive "already done" or "still running, no live progress until next event".
- **UX**: explicit reentry banner ("Welcome back — checking your generation…") followed by either result reveal or live progress resumption.
