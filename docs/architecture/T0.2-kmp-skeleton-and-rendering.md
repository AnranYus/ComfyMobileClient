### 模块划分原则
- **`:shared`（KMP 模块）**：包含所有业务、数据、协议、UI（用 Compose Multiplatform）；目标 99% 代码在 `commonMain`。
- **`:androidApp` / iosApp**：薄壳。Android 用单 Activity + ComposeRoot；iOS 用 `Main.kt` 暴露 `MainViewController()` 给 SwiftUI。
- **`platform/` 模块内的 expect/actual**：仅用于真正必须分平台的能力（网络栈监听、文件路径、低级 IO）。决不为业务逻辑写 expect。

### Compose Multiplatform vs 拆分 Compose+SwiftUI 的取舍
- **选 Compose Multiplatform**（默认）：UI 一份，节点图渲染同样跨端，开发速度最快。Compose iOS 已稳定（2026 视角）。
- **拆分 Compose（Android）+ SwiftUI（iOS）**（备选）：仅在 T0.3 硬指标证明 iOS 上 Skiko 性能不足以支撑节点图时启用。届时 `:shared` 仅保留 commonMain 业务层，UI 各写一遍。
- 即便采用 Compose Multiplatform，也要把节点图渲染抽到独立 Composable 包（`presentation/editor/graph/`），保持后续可替换性。

### 状态管理选择
**推荐：MVI + Compose State (StateFlow)**，无第三方框架。
- `commonMain` 里每个屏幕一个 `XxxViewModel`（继承共享 `ViewModel` 抽象，不强依赖 androidx.lifecycle 的 ViewModel — 用 `koinScope` + 协程 scope 自管理生命周期）。
- 状态 = `StateFlow<XxxState>`，事件 = `XxxIntent` sealed class。
- 不引入 Decompose（导航复杂度不需要） / Voyager（耦合 Android）/ MVIKotlin（额外抽象）。Compose Multiplatform Navigation 已可用。
- 例外：若 T0.3 信息架构出现复杂多层导航，再回头评估 Decompose。

### 序列化 / 持久化
- **kotlinx.serialization** —— JSON、CBOR 备选。Workflow JSON 用 `JsonElement` 保留未知字段（关键：roundtrip 必须）。
- **SQLDelight** —— 跨端 SQLite，给历史记录、缓存。
- **Multiplatform Settings (Russhwolf/multiplatform-settings)** —— 给"上次连过的 IP"等小配置。

### 网络
- **Ktor Client Core** + 平台 engine：
  - Android: `OkHttp` engine
  - iOS: `Darwin` engine
- **Ktor Client WebSockets** plugin —— 提供 `client.webSocket { ... }` 协程化 API。
- **Ktor Client ContentNegotiation + kotlinx-serialization-json** —— 自动 JSON 编解码。

### 协程 / 调度
- **kotlinx.coroutines** —— 标配。
- 每个 ViewModel 持有自己的 `CoroutineScope(SupervisorJob() + Dispatchers.Main.immediate)`，在生命周期销毁时 cancel。
- 网络层全部 IO Dispatcher。

### 图片加载与缓存
- **Coil 3 (Multiplatform)** —— 直接支持 Compose Multiplatform，magnet。给画廊与 PreviewImage 缩略图。
- 自定义 `Fetcher` 适配 ComfyUI `/view` URL 构造。

### Workflow 数据模型在 commonMain 的早期形态

```kotlin
// domain/workflow/Workflow.kt（草案）
@Serializable
data class WorkflowEnvelope(
    val original: JsonElement,        // 原 import 的 JSON 不动
    val format: WorkflowFormat,       // UI | API
    val metadata: WorkflowMetadata,   // 本地标签、导入时间、最近编辑
)

@Serializable
sealed class WorkflowGraph {
    @Serializable @SerialName("ui")  data class Ui(val raw: JsonObject) : WorkflowGraph()
    @Serializable @SerialName("api") data class Api(val nodes: Map<String, ApiNode>) : WorkflowGraph()
}

@Serializable
data class ApiNode(
    @SerialName("class_type") val classType: String,
    val inputs: Map<String, JsonElement>,  // 字面量 or [src_id, idx] 用 JsonElement 保留
)
```

**关键设计**：所有 unknown 字段都 `JsonElement` 保留，不强类型化，roundtrip 安全。

## M2 — 节点图渲染层候选矩阵  *(Status: ⏳ awaiting @Ores T0.3 hard-requirements draft)*

### 候选 A：纯 Compose Canvas
- 实现：`Canvas { drawXxx() }` + 自管 scale/translate matrix；节点用 `Box` 或 `Layout` 内嵌 Compose 控件做参数 picker（命中区域用 Modifier.pointerInput 处理手势）。
- 优势：开发简单、参数控件天然是 Compose 控件、热重载快。
- 风险：~ 30 节点以下流畅；超出后 Compose 重组成本上升，pan/zoom 60fps 不易保证；不支持精细化 GPU pipeline。

### 候选 B：Skiko 直绘
- 实现：在一个 Composable 内拿 `DrawScope.drawIntoCanvas { canvas: org.jetbrains.skia.Canvas -> ... }` 直接调用 Skia API（Path / Paint / Picture），节点的"绘制"层用 Skia，参数控件层仍是 Compose 浮层。
- 优势：性能上限明显高于纯 Compose Canvas；可使用 Skia Picture 缓存静态部分；连线绘制（贝塞尔）成本低。
- 风险：参数控件与图形层组合对齐复杂；命中检测需要自管空间索引；调试工具偏弱。

### 候选 C：平台分叉
- 实现：Android 走候选 A 或 B；iOS 用 SwiftUI + GraphicsContext（或 Metal）单独写。`:shared` 只到 ViewModel/Domain。
- 优势：iOS 上 Skiko 不行的兜底方案。
- 代价：UI 工程量翻倍；T0.2 输出一份"如何分叉"的指南。

### 评分维度（待 T0.3 硬指标到位后填）

| 硬指标 | A 纯 Compose Canvas | B Skiko 直绘 | C 平台分叉 |
|---|---|---|---|
| 同屏节点 ≤ 30 | TBD | TBD | TBD |
| 双指缩放 + 60fps 拖动 | TBD | TBD | TBD |
| 节点参数控件混合（滑块/dropdown/多行 prompt） | TBD | TBD | TBD |
| 连线绘制（贝塞尔，>50 连线） | TBD | TBD | TBD |
| 抽屉打开 < 200ms | TBD | TBD | TBD |
| 工程量 / 工时 | 1× | 1.5× | 2× |
| iOS 风险 | 低 | 中（Skiko 在 iOS 是 Compose 内部使用，可行但需 tunin） | 低（用原生） |
