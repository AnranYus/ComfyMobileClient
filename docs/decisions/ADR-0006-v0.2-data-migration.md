# ADR-0006: v0.2 data migration — `workflow.imported_original_json` + first multi-statement `.sqm`

- Date: 2026-06-02
- Status: Proposed
- Decided in: Slock #ComfyMobile (msg `aec273e6` ADR-0006 commission). Forward-references [ADR-0005](ADR-0005-topology-mutation.md) §2 / §3 / §Consequences "Schema deferred" item.
- Owner (decision): @Priestess (project host), @Cody (main dev — migration + `.sq` head schema), @Longly (this ADR + migration test harness)
- Related research: ADR-0005 §2 + §3 (consumer side); thread `#ComfyMobile:7c615a2a` for Cody alignment on multi-statement `.sqm` + test fixture availability.

## Context

ADR-0005 introduces `WorkingGraph.resetToImported()` and a constructor parameter `importedOriginal: JsonElement` (non-nullable) that needs a persisted source. The required source is a per-row column on the SQLDelight `workflow` table, immutable for the row's lifetime, equal to the as-imported JSON.

The current schema (`shared/src/commonMain/sqldelight/com/comfymobile/db/Workflow.sq` head + `1.sqm`..`3.sqm` migrations) gives `workflow` these columns:

```
id TEXT PRIMARY KEY, friendly_name TEXT, format TEXT, original_json TEXT,
metadata_json TEXT, imported_at INTEGER, last_opened_at INTEGER, last_run_prompt_id TEXT
```

`original_json` was structurally lossless under [ADR-0003](ADR-0003-workflow-roundtrip-strategy.md) **only because nothing wrote to it after import**. ADR-0005's `confirm()` is the first writer, which means `original_json` after Phase 3 is "current head", not "as-imported". The rollback anchor needs a new immutable column.

Two upstream constraints shape the migration:

1. **ADR-0005 §3 ctor `importedOriginal: JsonElement` is non-nullable.** A nullable column on the SQL side would force one of: a non-null fallback (which has no defensible value), a load-time `requireNotNull` (which moves the failure mode from migration to first read of an old row), or weakening ADR-0005's ctor contract. None acceptable.
2. **The existing `1.sqm`/`2.sqm`/`3.sqm` are all single `ALTER TABLE`.** Adding a non-nullable column to SQLite cannot be done with a single `ALTER` (SQLite rejects `ALTER TABLE ... ADD COLUMN ... NOT NULL` without a non-NULL default; the only well-defined non-NULL default is `original_json` itself, which is dynamic per row). The standard SQLite portable pattern is table-rebuild: rename → create-new → insert-select → drop → recreate-indexes. This ADR is the project's first multi-statement `.sqm`.

There is no existing SQLDelight migration test infrastructure in `shared/src/commonTest`. SQLDelight 2.0.2 ships built-in support for `verifyMigrations(true)` + `schemaOutputDirectory`, which has not been wired into `shared/build.gradle.kts`.

## Decision

### 1. New column

`workflow` gains:

```
imported_original_json TEXT NOT NULL
```

Semantics: byte-identical to the JSON that `WorkflowImporter` produced on the import that created this row. Never mutated after row creation. Used exclusively by `WorkingGraph.resetToImported()` (ADR-0005 §3).

### 2. Head-schema edit (`Workflow.sq`)

The `CREATE TABLE workflow` block at the top of `Workflow.sq` adds the column in declaration order between `original_json` and `metadata_json`. The `upsertWorkflow` query gains one positional parameter in the same column position. All other queries (`selectWorkflowById`, `selectAllWorkflows`, `selectRecentWorkflows`, `markWorkflowOpened`, `renameWorkflow`, `deleteWorkflow`) need no change — `SELECT *` picks up the new column automatically; the targeted UPDATEs / DELETEs don't reference it.

### 3. `4.sqm` migration — first multi-statement table rebuild

```sql
-- 4.sqm — adds workflow.imported_original_json (ADR-0006).
-- SQLite cannot ALTER TABLE ... ADD COLUMN ... NOT NULL without a
-- non-NULL default. The defensible default for existing rows is the
-- per-row original_json: pre-Phase-3 there was no writer for
-- original_json, so for every existing row original_json equals the
-- JSON WorkflowImporter produced at row creation; ADR-0003's lossless
-- invariant still holds at the backfill instant, which is what makes
-- imported_original_json := original_json the right migration.
-- Standard portable rebuild pattern: rename + create + insert-select +
-- drop + recreate indexes.

ALTER TABLE workflow RENAME TO workflow_v3;

CREATE TABLE workflow (
    id TEXT NOT NULL PRIMARY KEY,
    friendly_name TEXT NOT NULL,
    format TEXT NOT NULL,
    original_json TEXT NOT NULL,
    imported_original_json TEXT NOT NULL,
    metadata_json TEXT NOT NULL,
    imported_at INTEGER NOT NULL,
    last_opened_at INTEGER,
    last_run_prompt_id TEXT
);

INSERT INTO workflow(
    id, friendly_name, format,
    original_json, imported_original_json,
    metadata_json, imported_at, last_opened_at, last_run_prompt_id
)
SELECT
    id, friendly_name, format,
    original_json, original_json,
    metadata_json, imported_at, last_opened_at, last_run_prompt_id
FROM workflow_v3;

DROP TABLE workflow_v3;

CREATE INDEX workflow_recents
    ON workflow(last_opened_at DESC, imported_at DESC);
```

Backfill rule: `imported_original_json := original_json`. Justification: pre-Phase-3 rows were never edited (no writer existed for `original_json`), so by ADR-0003 the two are byte-identical for any row that exists at migration time. After migration, only fresh imports set both columns and only `confirm()` mutates `original_json`; the invariant holds going forward.

### 4. Importer-side change

`WorkflowImporter` (the only producer of `workflow` rows) writes `imported_original_json` = `original_json` on every `upsertWorkflow` for a new row. The two-column write must be atomic — both populate from the same parsed JSON in the same statement.

For repeat-import of an existing `id` (re-importing the same workflow file): `imported_original_json` of the existing row is **not** overwritten. The semantics is "this column captures the *first* import of this row". A re-import that wants to reset to the new file content is the user invoking "delete then re-import"; we don't quietly rewrite the rollback anchor on subsequent imports because that would break the user's mental model of "reset to imported = the import I did". This is an importer-side guard:

```kotlin
val existing = workflowQueries.selectWorkflowById(id).executeAsOneOrNull()
val importedOriginal = existing?.imported_original_json ?: rawJson
workflowQueries.upsertWorkflow(
    id, name, format, rawJson, importedOriginal, metadata, now, openedAt, lastRunId,
)
```

The read-then-upsert sequence is **safe under the current single-threaded import path** — `WorkflowImporter` is the only producer of `workflow` rows, and Phase 2 routes all imports through a single coroutine. Any future parallel-import path (e.g. batch-import from share-sheet multi-file) must wrap this sequence in a transaction; flag in the consumer PR at that time.

### 5. Build / CI wiring — first migration verification

`shared/build.gradle.kts` `sqldelight { databases { create("ComfyMobileDb") { … } } }` block gains:

```kotlin
verifyMigrations.set(true)
schemaOutputDirectory.set(file("src/commonMain/sqldelight/databases"))
```

`verifyMigrations` makes SQLDelight's code-gen task fail when applying `1.sqm`..`4.sqm` in sequence to an empty database produces a schema that diverges from the `.sq`-declared head schema. The committed `databases/4.db` (auto-generated golden schema dump produced by `:shared:generateSqlDelightSchema`) is the diff target — the T3.1 PR commits it alongside `4.sqm` and `Workflow.sq`. CI (`Android build smoke`) covers this transitively because `assembleDebug` depends on `generateCommonMainComfyMobileDbInterface`, which depends on `verifySqlDelightMigration`.

### 6. Test coverage in `androidUnitTest`

#### 6a. New shared helper — in-memory driver factory

This codebase has no prior `ComfyMobileDb` in-memory test setup (per thread `#ComfyMobile:7c615a2a` Q2). One small helper underpins both the migration test and any future SQLDelight-touching test:

```kotlin
// shared/src/androidUnitTest/kotlin/com/comfymobile/data/db/InMemoryComfyMobileDb.kt
internal fun newInMemoryComfyMobileDb(): ComfyMobileDb {
    val driver = JdbcSqliteDriver(JdbcSqliteDriver.IN_MEMORY)
    ComfyMobileDb.Schema.create(driver)
    return ComfyMobileDb(driver)
}
```

Lives in `androidUnitTest` (the JVM-on-Android unit-test source-set @Cody opened in PR #45 for the `AndroidLifecycleMonitor` regression test), not `commonTest`. `JdbcSqliteDriver` is JVM-only; placing it in `commonTest` would compile fine today but become a hidden landmine the first time CI starts running `:shared:iosArm64Test` or any other native target's test compile. `androidUnitTest` is the precise scope: JVM execution, but never a `commonTest` cross-platform claim. Migration semantics under SQLite are portable across drivers; if a future need arises to verify migration on the iOS native driver, add an `iosTest` mirror at that time.

Requires `app.cash.sqldelight:sqlite-driver` in the `androidUnitTest` source-set dependencies (added alongside the existing `kotlin("test")` + `kotlinx-coroutines-test` from PR #45's `androidUnitTest by getting { dependencies { … } }` block).

#### 6b. `Migration4Test`

New file: `shared/src/androidUnitTest/kotlin/com/comfymobile/data/workflow/Migration4Test.kt`. Three cases:

| # | Setup | Assertion |
|---|---|---|
| M4-1 | New `JdbcSqliteDriver(IN_MEMORY)`; run `ComfyMobileDb.Schema.migrate(driver, 3, 4)`; query `PRAGMA table_info(workflow)` | `imported_original_json` present, `NOT NULL`, `TEXT`; column order matches head schema |
| M4-2 | Pre-load 3 rows at schema v3 via raw `driver.execute(...)` with varied `original_json` (small / large / contains-unicode); run migrate 3→4; `SELECT id, original_json, imported_original_json` | For every row, `imported_original_json == original_json` byte-identical; row count = 3 (no duplication, no loss); other columns (friendly_name, format, metadata_json, imported_at, last_opened_at, last_run_prompt_id) preserved |
| M4-3 | Migrate empty 3→4; insert a fresh row via the v4 head `upsertWorkflow` query with `imported_original_json := original_json` (the importer-shape call); `selectWorkflowById` | Insert succeeds; row returns with both JSON columns populated and equal |

The `Migration4Test` doubles as proof that SQLDelight 2.0.2 + bundled JVM SQLite driver execute multi-statement `.sqm` correctly (see §Consequences).

## Alternatives considered

- **Nullable column** (`imported_original_json TEXT`, no NOT NULL). Rejected. ADR-0005 §3 ctor takes `importedOriginal: JsonElement` (non-nullable); a nullable column forces either weakening that contract, a `requireNotNull` at first read (moves failure from migration time to runtime), or a kotlin-side fallback with no defensible value. The whole point of doing the migration is to make the new invariant unconditional from Kotlin's perspective.
- **Single `ALTER TABLE workflow ADD COLUMN imported_original_json TEXT NOT NULL DEFAULT '{}'`.** Rejected. The empty-object literal is not a valid import baseline — ADR-0003's lossless invariant attaches to the per-row JSON, and `{}` collapses every existing row's rollback anchor to the same syntactic minimum. A user who hits "reset to imported" on a pre-Phase-3 row would lose their workflow entirely.
- **Drop and recreate the table without preserving rows.** Rejected. Destroys all user-imported workflows. Even at v0.1.x scale (1 known user) this is the wrong default — migration semantics are a forward-compatible contract.
- **Defer the column until the first `confirm()` writes** (lazy init via kotlin-side branch on null). Rejected. Couples ADR-0005's editor code to migration state; every read of `importedOriginal` would need a null check; "reset to imported" semantics for rows that haven't been confirmed yet would silently fall back to `original_json`, which is correct-but-confusing on the day of migration and incorrect afterwards.
- **SQLite 3.25+ `ALTER TABLE ADD COLUMN` + later `ALTER` to NOT NULL.** Rejected. SQLite does not support promoting an existing column to NOT NULL in place — only the rebuild path does it portably. Plus android-driver / native-driver ship varying SQLite versions and we can't pin a floor without a separate ADR.

## Consequences

- Pro: ADR-0005 §3 ctor stays non-nullable; the editor never has to ask "did this row predate Phase 3?" — every row at v4 has a valid rollback anchor by construction.
- Pro: Establishes the project's table-rebuild pattern (rename / create / insert-select / drop / recreate-index) for future non-nullable column additions. Documented here so future ADRs can reference one pattern instead of relitigating.
- Pro: `verifyMigrations(true)` + schema dump gives every future migration automatic divergence detection — the next time someone edits `.sq` without writing a matching `.sqm` (or vice versa), the build fails. Closes a class of bug we've been one schema-edit away from since `1.sqm`.
- Pro: Migration test harness (`Migration4Test`) and the JVM in-memory driver setup are reusable for `5.sqm`+; this ADR pays for the harness once.
- Con: Table rebuild rewrites every existing `workflow` row at upgrade time — O(N) IO. At v0.1.x scale (single-digit users, <1000 rows) negligible; future schema changes touching `workflow` should still prefer in-place `ALTER` if a non-null default exists.
- Con: Adds one new dependency in `androidUnitTest` (`sqldelight:sqlite-driver`) and one CI verification step. Build time impact expected <1s for the migration verify.
- Con: First multi-statement `.sqm` in the codebase — until `Migration4Test` runs in CI we have no automated proof SQLDelight 2.0.2 + the bundled SQLite driver execute multi-statement `.sqm` correctly. The test is the proof. **Roll-forward fallback**: if multi-statement execution fails under some driver, split into `4.sqm` (rename + create + insert-select) and `4b.sqm` (drop old + recreate-index) using SQLDelight's stable migration-numbering pattern; the head schema in `Workflow.sq` is unchanged either way.
- Con: No foreign-key columns reference `workflow.id` today (confirmed in thread `#ComfyMobile:7c615a2a` Q3 — `Job.sq:workflow_id` is bare, no `FOREIGN KEY` clause), so the rebuild does not need `PRAGMA foreign_keys = OFF`. If a future schema change introduces an FK pointing at `workflow.id`, that migration must add the pragma toggle around any subsequent table rebuild — call out in the consumer ADR at that time.

## Amendments

### A1 (2026-06-02): `verifyMigrations(true)` deferred until bootstrap baselines exist

Surfaced during T3.1 implementation (PR thread msg `9bac0273`): flipping `verifyMigrations.set(true)` in `shared/build.gradle.kts` immediately fails the build because Phase 2's `1.sqm` / `2.sqm` / `3.sqm` are all bare `ALTER TABLE` statements that assume the prior `vN-1` schema already exists in the database. SQLDelight's `verifyMigrations` runs each `.sqm` against an **empty** database (or against a committed `databases/(N-1).db` baseline when present) and asserts the resulting schema equals the head `.sq`. With no committed baselines for v0..v3, every migration in the Phase 2 chain fails on first contact — `1.sqm` cannot `ALTER TABLE jobIndex` against a database that has never had a `jobIndex` table created.

`4.sqm` itself surfaces the same class of issue independently: the head `.sq` already declares `CREATE INDEX workflow_recents`, and the migration's own `CREATE INDEX workflow_recents` at the end of the table rebuild produces a `Duplicate index name workflow_recents` collision when re-applied against the head schema.

Three resolution paths were considered:

- **(A) Generate `databases/0.db` .. `databases/3.db` baselines now**, by git-time-traveling `Workflow.sq` to each prior commit + running `Schema.create` per snapshot. Standard SQLDelight pattern. Rejected for the T3.1 scope: requires reconstructing Phase 2 intermediate schemas (not just commit IDs — the actual snapshotted `Workflow.sq` shape at the moment each `.sqm` was authored), which is archaeology that doesn't belong in the migration-introducing PR.
- **(B) Rewrite `1.sqm` / `2.sqm` / `3.sqm` so they're executable against an empty database** (e.g. `CREATE TABLE IF NOT EXISTS jobIndex (...full v1 shape...)` instead of bare `ALTER`). Rejected. Changes the semantics of already-deployed migrations — any v0.1.x user with a v3 SQLite DB on their device would receive a different upgrade path than the one that originally produced their schema. @Nothing's OPLUS device is exactly such a deployed v3 DB; v0.2 cannot risk a schema fork there.
- **(C) Defer `verifyMigrations`. Rely on `Migration4Test` as the explicit proof of the 3→4 migration. Keep `schemaOutputDirectory.set(...)` enabled — the schema dump still produces `databases/4.db` as a golden artifact even with verification off, which becomes a reference point for the eventual A path.** Adopted.

What this amendment changes in §5:

- `verifyMigrations.set(true)` is **NOT** wired into `shared/build.gradle.kts` in T3.1.
- `schemaOutputDirectory.set(...)` **is** wired (independent of verifyMigrations); T3.1 commits `databases/4.db` as the head schema dump.
- The "automatic divergence detection" Pro listed in §Consequences is correspondingly downgraded for the v0.2 window: `Migration4Test`'s three explicit cases (schema shape via `PRAGMA table_info`, backfill correctness on 3 varied rows, post-migration insert via the v4 `upsertWorkflow`) are the divergence proof for this single migration.

Tracked follow-up: task **v0.2 chore — schema bootstrap baselines + re-enable verifyMigrations** (project task #17). When that PR lands, `verifyMigrations(true)` becomes the standing forward-detection mechanism for `5.sqm`+; until then, every new `.sqm` ships with its own explicit `MigrationNTest`.

The roll-forward fallback for multi-statement `.sqm` execution (§Consequences) is unaffected — if `Migration4Test` reports that SQLDelight 2.0.2 + the bundled SQLite driver can't execute multi-statement `.sqm`, the `4.sqm`/`4b.sqm` split path still applies regardless of whether `verifyMigrations` is on.
