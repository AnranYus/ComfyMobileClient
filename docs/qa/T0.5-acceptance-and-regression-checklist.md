## 0. 当前产品边界

- 用户：取用型轻度用户，主路径为“导入工作流 -> 调关键参数 -> 生图 -> 查看产物/历史”。
- 设备：手机为主，兼顾平板响应式。
- 技术：KMP / Compose Multiplatform。
- 后端：同一局域网内连接用户自建 ComfyUI。
- 连接：MVP 仅手动输入 IP:端口 + 最近连接历史；不做 mDNS/Bonjour 自动发现。
- 工作流：移动端允许简化视图，但原始 workflow JSON 必须完整保留，roundtrip 不丢信息。
- 编辑：白名单节点通过数据驱动的“节点描述符表”暴露参数；白名单外节点默认只读，参数和拓扑均不可编辑。UX 需提供“重置到导入时”恢复入口；用户尝试编辑 unsupported node 时给出明确提示。

## 1. 参考事实

- ComfyUI 官方 Server Routes 包含 `/prompt`、`/ws`、`/view`、`/object_info`、`/history`、`/queue`、`/interrupt` 等核心接口；`/prompt` 负责提交并进入执行队列。
- `/ws` 承载执行状态、节点执行、进度、错误与队列变化等实时消息；实际事件粒度已由 T0.1 确认。
- Workflow JSON 有正式 schema，包含 nodes、links、groups、reroutes、layout/state、widgets_values 以及扩展字段；移动端必须保留未知字段。
- Node Definition JSON 有 inputs/outputs、类型、display hints、tooltip、image upload 等字段，适合作为节点描述符表和参数校验依据。
- T0.1 M2 明确：UI workflow JSON 不能直接提交给 `/prompt`，Mobile 必须做 UI -> API prompt 转换；生成时必须在 `extra_data.extra_pnginfo.workflow` 写入**提交时的 current UI-format workflow snapshot after mobile edits**，确保 PNG 可回桌面 ComfyUI 恢复且包含本次移动端修改。
- T0.1 M2 明确：动态枚举值不应写死，`sampler_name` / `scheduler` 等来自 `/object_info`，模型列表来自 `/models/{folder}`，embedding 列表来自 `/embeddings`。

来源：
- https://docs.comfy.org/development/comfyui-server/comms_routes
- https://docs.comfy.org/specs/workflow_json
- https://docs.comfy.org/specs/nodedef_json

## 3. 需求 1：导入工作流

功能边界：
- 支持导入 ComfyUI workflow 文件，完整保留原始 JSON。
- 必须识别 UI workflow JSON 与 API prompt JSON 的差异；若二者都支持，UI 与数据模型必须显式区分。
- 可以先本地解析/预览；运行和节点能力校验依赖 LAN ComfyUI 时，需要明确提示。

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| I-01 | 基础 workflow 可导入 | 导入基础文生图 workflow | 显示节点概览、关键参数入口、工作流名称/摘要 |
| I-02 | 未知字段保留 | 导入含 custom fields、reroutes、groups、widgets_values 的 workflow 后导出 | 未知字段和图结构结构等价；无静默删除 |
| I-03 | 自定义节点不阻塞预览 | 导入含未知 custom node 的 workflow | 未知节点可见、参数只读、整体预览不崩溃 |
| I-04 | 缺失依赖可解释 | 连接 LAN ComfyUI 后导入缺模型/缺自定义节点样例 | 错误能区分缺模型、缺节点、JSON 损坏、版本不支持 |
| I-05 | 格式混淆有保护 | 分别导入 UI workflow JSON 与 API prompt JSON | 系统识别格式；不把 UI workflow 直接误提交为 API prompt |
| I-06 | 大图不崩溃 | 导入 20+ 节点工作流；如 T0.3 要求则加 50+ 节点 | 手机/平板不崩溃，关键操作不被遮挡 |
| I-07 | widget 参数映射可验证 | 导入含 `widgets_values[]` 的白名单节点 workflow 并转换 API prompt | 参数按 `/object_info[node.type].input.required` 顺序或白名单 fallback 正确映射 |

主要风险：
- UI workflow JSON 与 API prompt JSON 混用导致“能导入但不能运行”。
- 导入时重排或删掉未知字段，破坏桌面 ComfyUI roundtrip。
- 缺模型/缺 custom node 错误过于技术化，轻度用户无法处理。

## 4. 需求 2：编辑工作流

功能边界：
- MVP 采用“完整只读 + 简化可编辑”：保存完整原始 workflow，移动端只暴露白名单参数。
- 白名单来自数据驱动节点描述符表，不写死在业务逻辑里。
- 白名单外节点参数只读；若后续允许删节点/改连线，必须增加确认、撤销、运行前校验。

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| E-01 | 编辑范围最小化 | 修改 prompt / sampler / size 等白名单参数后做 JSON diff | 只修改目标节点目标字段；其他字段保持不变 |
| E-02 | 参数类型校验 | 对 INT/FLOAT/BOOLEAN/COMBO/STRING/IMAGE 参数输入非法值 | 提交前本地拦截并给出字段级错误 |
| E-03 | prompt 移动端可用 | 长 prompt 粘贴、软键盘弹出、横竖屏变化 | 输入框、保存、取消、关闭操作不被遮挡 |
| E-04 | 可回退 | 修改多个参数后点放弃/恢复原值 | 回到编辑前状态，JSON diff 为空 |
| E-05 | 跨端序列化一致 | Android/iOS 对同一参数做同一编辑 | 输出 JSON diff 一致 |
| E-06 | ComfyUI 校验可解释 | 编辑后提交到 LAN ComfyUI，触发 `/prompt` validation error | 能将 node_errors 映射到节点/参数/用户可读提示 |
| E-07 | 动态选项不写死 | 更换模型目录或 sampler/scheduler 列表后打开编辑 UI | 下拉/模型选择来自 `/object_info`、`/models/{folder}`、`/embeddings` 等运行时来源 |
| E-08 | 重置入口可达 | 工作流视图右上角菜单触发重置 | 明确二次确认；确认后恢复导入态，取消则不改动 |
| E-09 | 轻度用户参数步进可用 | 测试 cfg、denoise、steps、width/height、seed、LoRA strength 控件 | cfg step 0.5、denoise step 0.05、steps 有 Quick=20/Default=30/High=50 预设，尺寸有 512/768/1024/1536 预设，seed 有随机与锁定入口 |

主要风险：
- 简化视图编辑破坏完整 workflow，桌面无法打开或运行。
- 控件类型和节点定义不匹配，例如枚举、图片、数字范围处理错误。
- Android/iOS 序列化差异导致同一编辑结果不同。

## 5. 需求 3：增/减/编辑节点

功能边界：
- 当前方向下，建议 MVP 将“编辑节点”定义为白名单节点参数编辑，不把自由拓扑编辑作为 Phase 2 阻塞项。
- 如 Phase 3 允许新增、删除、重连，需要先定义安全边界：哪些节点可加、哪些节点可删、哪些边可改、未知节点如何保护。

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| N-01 | 节点描述符表驱动 | 修改描述符表中某节点的显示名/控件类型/可编辑参数 | UI 按配置变化，无需改业务代码 |
| N-02 | 未知节点保护 | 导入含未知节点工作流，尝试编辑参数或拓扑 | 参数只读、不可删除、不可改连线；不会生成错误 JSON |
| N-03 | 新增节点受控 | 尝试添加非白名单节点 | MVP 中不可添加，或显示“暂不支持”；不允许任意 custom node |
| N-04 | 删除/重连安全 | 若启用删节点/改连线，删除白名单节点或改边后运行前校验 | 无不可见悬空边；类型不匹配能定位到具体 link/node |
| N-05 | 手势不冲突 | 手机上执行画布缩放/平移、节点选择、抽屉滚动、文本输入 | 手势互不抢焦点；不会误拖线或误删节点 |
| N-06 | 平板一致 | 平板侧栏编辑与手机抽屉编辑同一节点 | 行为和 JSON diff 一致 |
| N-07 | 解锁拓扑编辑有门槛 | 若后续允许 unknown node 删除/改连线 | 必须同时具备用户确认、撤销可达、运行前类型/连通性校验 |
| N-08 | Unsupported 操作提示 | 对 unknown node 执行删除/改线/参数编辑手势 | 操作被拦截；toast 提示“此节点暂不支持移动端编辑，可在桌面 ComfyUI 中调整后重新导入” |
| N-09 | 运行态禁用危险手势 | 生成运行中尝试拖动节点、改连线、改参数、重置工作流 | 操作被禁用或进入只读态；不影响正在运行的 `prompt_id` 和 workflow snapshot |

主要风险：
- 允许删除/重连未知节点会破坏 custom node workflow，且轻度用户难以理解后果。
- 节点描述符表不跟随 ComfyUI/custom node 能力演进。
- 触摸手势冲突导致误操作。

## 6. 需求 4：获取生图产物

功能边界：
- 提交、监听、取消、拉取产物围绕 `prompt_id` 建立状态机。
- 已确认核心协议：`/prompt`、`/ws`、`/history/{prompt_id}`、`/view`、`/interrupt`；运行中的取消使用 `/interrupt`，队列中未开始的任务才使用 `/queue` 删除。
- T0.4 连接态按 3 分支验收：A 稳定 WebSocket、B 前台 LAN 抖动、C 后台挂起恢复；B 与 C 不合并，因为 OS 后台挂起会丢 WebSocket 事件，语义不同。

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| O-01 | 提交记录完整 | 提交一次生成 | 记录 `prompt_id`、workflow 快照、关键参数、服务器、提交时间 |
| O-02 | 状态可恢复 | 生成中锁屏/后台/切 Wi-Fi/断开 WebSocket 后返回 | 能通过队列/历史恢复状态，或明确告知无法恢复 |
| O-03 | 重试不重复 | 提交后网络超时并点击重试 | 不重复排队；若状态未知，必须让用户确认 |
| O-04 | 状态分类清楚 | 模拟排队、运行、完成、失败、中断、连接中断 | UI 状态不同且文案可理解 |
| O-05 | 中断不误判 | 运行中触发取消并调用 `/interrupt`；另测未开始队列项通过 `/queue` 删除 | 历史状态标为中断/取消或已移除，不把部分产物当完整成功；两种路径不混用 |
| O-06 | 产物归属正确 | 两个任务连续提交并交错完成 | 输出图、缩略图、参数、`prompt_id` 不串 |
| O-07 | 单图拉取可重试 | 多图任务中让其中一张 `/view` 失败 | 支持单张重试，不要求重新生图 |
| O-08 | 输出嵌入 workflow | 提交生成时检查请求与生成 PNG 元数据 | 请求包含 `extra_data.extra_pnginfo.workflow = current UI-format workflow snapshot after mobile edits`，PNG 可回桌面恢复且包含用户本次修改 |
| O-09 | 三类连接态分别可恢复 | 分别测试稳定 WS、前台 LAN 抖动、后台挂起 30s+ | A 显示连续进度；B 进入连接抖动/重连但保留任务；C 回前台后探活并用 queue/history 恢复 |
| O-10 | 恢复后完成态不幽灵挂起 | 在 B 前台 LAN 抖动和 C 后台挂起期间让 prompt 实际完成，恢复后刷新状态 | 已完成任务标记为成功并可查看产物；不会残留幽灵 `in_progress` |
| O-11 | B 分支不制造警报感 | 前台 LAN 抖动 <=5s 后恢复 | 保留最后已知进度并静默显示重连/心跳信任态；不弹显式 alarm banner |
| O-12 | C 分支回归 banner 可观察 | 后台挂起 30s+ 后回前台 | 显示“欢迎回来，正在检查…”类 banner，并通过 `/history` 一次性补查结果 |
| O-13 | 长时间重连进入 Lost | 前台 LAN 抖动后 30s 仍未恢复 | 进入 Lost/可恢复错误态，保留任务记录和重试/检查入口 |
| O-14 | ETA 不跳变 | A -> B -> A 或 B/C 恢复时观察 ETA | ETA 使用容差/降级文案，不出现明显反复跳变或精度伪装 |

主要风险：
- WebSocket 断开后任务仍在跑，但移动端误判失败或重复提交。
- 队列中多个任务产物串图。
- 大图拉取造成内存压力、缩略图卡顿或崩溃。

## 7. 需求 5：查看历史产物

功能边界：
- 历史至少包含本机提交过的任务；是否浏览 ComfyUI 全局历史由 T0.1/T0.3 决定。
- 历史保存 workflow 快照、关键参数、产物引用、服务器身份、状态和时间。

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| H-01 | 历史项可追溯 | 完成生成后进入历史详情 | 显示产物、时间、状态、prompt/参数摘要、服务器、workflow 快照 |
| H-02 | 服务端产物失效可降级 | 清理 ComfyUI 历史或删除输出文件后打开历史 | 显示“记录存在，产物不可取回”，不崩溃不空白 |
| H-03 | 多服务器不混淆 | 连接两个不同 LAN ComfyUI 地址并分别生成 | 历史按服务器区分，不混入旧实例产物 |
| H-04 | 再次运行不污染当前编辑 | 从历史点击再次运行，同时当前有未保存编辑 | 使用历史 workflow 快照，不覆盖当前编辑态 |
| H-05 | 缓存可控 | 生成多批大图后检查缓存 | 有缓存上限、清理策略或系统压力下退化策略 |
| H-06 | 缩略图与原图一致 | 打开历史列表和详情图 | 缩略图对应同一产物文件，不串图 |

主要风险：
- ComfyUI 服务端历史被清理后，移动端还持有失效引用。
- 同 IP 变更设备或 DHCP 变化导致历史归属混淆。
- 再次运行历史任务污染当前编辑态。

## 8. LAN 手动连接专项

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| L-01 | IP:端口格式校验 | 输入空值、中文、缺端口、非法 IP、端口越界、域名误填 | 本地即时提示，不发起无意义连接 |

| L-09 | 恢复补查完成态 | 前台抖动/后台挂起期间服务端已完成 prompt | 客户端通过 history/view 补查后进入成功态，不继续显示运行中 |
| L-10 | B/C 共享状态但 UX 分化 | 触发 `ReconnectReason.LAN_FLAKE` 和 `ReconnectReason.BACKGROUND_RESUMED` | 底层可共用 Reconnecting 状态；前者静默心跳，后者显示回归 banner |

说明：mDNS/Bonjour 自动发现已被 @nothing 明确排除在 MVP 之外，不列入 T0.5 MVP 回归。

## 9. KMP / 跨端专项

| ID | 规则 | 测试方法 | 预期 |
| --- | --- | --- | --- |
| K-01 | shared 层负责核心逻辑 | 检查实现或设计文档 | workflow 解析、diff、保存、提交前校验位于 shared/common 层 |

## 12. 最高优先级风险

1. Workflow 格式混淆：UI workflow 与 API prompt 混用会直接导致导入能看但无法运行；UI -> API 转换和 `widgets_values[]` 映射必须重点测试。
2. 简化编辑丢信息：未知节点和 unknown fields 若被重写，或未写入 `extra_data.extra_pnginfo.workflow`，会破坏 PNG/桌面 roundtrip 和用户信任。
3. LAN 恢复不可靠：移动端休眠、切网、WebSocket 断开最容易造成状态错乱或重复提交。
4. 产物归属错误：一旦 `prompt_id`、history、view 文件映射不严谨，会出现“图不是这次生成的”的严重体验问题。
5. 白名单硬编码：节点生态会变化，编辑能力必须由可配置描述符表驱动。
